                                                                  前言

                                                                  在正则表达式的学习和使用过程中，相信很多人到都遇到过这种情况：正则表达式在测试环节明明是没问题的，但应用到实际环境中却不管用。出现这种情况，很大概率是正则的“方言”问题导致的——即，不同的正则引擎之间是存在语法差异的，你的测试工具和实际环境所使用的正则引擎并不一致。要避免这类问题，“方言意识”是不可或缺的。

                                                                  以 Quicker 中使用正则为例，我们需要有这样的意识：Quicker 正则对应的“方言”（flavor）为 C#(.NET) ，那么正则测试工具最好选择支持 C# (.NET) 的，测试之时要注意切换至这个语言，语法的查询也要选择与之对应的教程/文档。

                                                                  本文的的目的就是为 C# (.NET) 提供一个正则语法速查手册。相比于微软官方文档，这个速查手册更为精要、平易、贴心。

                                                                  🤔 .NET 正则的代理对问题

                                                                  由于 .NET 字符串是使用 UTF-16 编码的，无论是目标字符串还是用于构建正则表达式的字符串，实际上都是 16-bit 代码单元（2个字节）的序列。而代码单元和 Unicode 字符之间的关系并不总是一一对应的，“ . 匹配任意一个字符”其实更符合实际的描述是“ . 匹配任意一个代码单元”。

                                                                  BMP 平面（U+0000 到 U+FFFF）中的字符使用一个代码单元即可表示，可以认为代码单元与 Unicode 字符是一一对应的，比如“汉”字的 Unicode 码点为 U+6C49，UTF-16 编码为 0x6C49。

                                                                  而辅助平面（大于 U+FFFF）的字符则需要使用一对代码单元表示。以“𠀾”字为例，其码点为 U+2003E ，UTF-16 编码为 0xD840 0xDC3E 。

                                                                  . 匹配字符串“𠀾”，结果会是：成功找到两个匹配项，分别是 0xD840 和 0xDC3E。如果我们需要匹配“𠀾”字，正则表达式可以写成 \uD840\uDC3E 或者直接输入 𠀾 。

                                                                  当正则表达式中出现辅助平面的字符时，需要意识到它并不是视觉上看到的一个“字符”，而是一对“特殊字符”。举两个例子：

                                                                  • 𠀾+ 匹配的是“一个 0xD840 ，接着多个 0xDC3E”， 可不是连续多个“𠀾”字——后者对应的正则应该是 (𠀾)+ 。
                                                                  • [甭𠀾] 匹配的是“甭、0xD840、0xDC3E 之中的任意一个字符（代码单元）”——如果想要匹配“甭”字或“𠀾”字，正确的写法是 甭|\uD840\uDC3E 或 甭|𠀾 。

                                                                   

                                                                   

                                                                  字符类 Character Classes

                                                                   

                                                                  💡 字符类（character class），也叫字符集（character set），作用是匹配某个字符集合中的任意一个字符。可以分为两大类：预定义字符类和自定义字符类。预定义字符类是正则引擎预先定义好字符集合，例如 . \d \p{Sc} 等；自定义字符类则是由用户自定义的，表示方法是使用方括号包含一系列字符 [characters]。

                                                                   

                                                                  📖 究竟什么是“单词字符”？

                                                                  我们都知道 \w 匹配的是“单词字符”（word character），但具体什么字符是“单词字符”，估计很多人都是不甚了了的。其实这里的“单词字符”并非人们口头上所说的英文单词字符——实际上，\w 主要是为匹配编程语言中的标识符而设计的，word character 就是标识符允许使用的字符，而标识符（例如变量名）通常是由字母、数字以及下划线构成的。早期的编程语言，因为设计者和使用者大多数都是以英语为母语的，所以标识符只支持 ASCII 字符。后来随着编程的国际化，越来越多的编程语言的标识符支持使用非 ASCII 字符。因此，正则表达式的 \w 所定义的字符集往往是有两套标准的：ASCII 标准和 Unicode 标准。

                                                                  例如，Javascript 正则的 \w 是 ASCII 标准的，等价于 [a-zA-Z0-9_]。C# 正则的 \w 默认是 Unicode 标准的，等价于 [\p{L}\p{Mn}\p{Nd}\p{Pc}] ，其中包含的字符不止是 [a-zA-Z0-9_]，还包括了汉字、日文假名、希腊字母、全角数字……这是 Unicode 版本的字母、数字以及连接符号；如果你设置了正则选项 ECMAScript ，那么 \w 将只匹配 ASCII 的“单词字符”，等价于 [a-zA-Z0-9_]。

                                                                  除了 \w，\b \d \s 同样存在 ASCII 标准和 Unicode 标准。因此，当我们使用这几个正则元素时，需要注意当前环境的正则引擎默认是哪一套标准。

                                                                  量词 Quantifiers

                                                                   

                                                                  😱 .* 的陷阱

                                                                  {0} 是一个特殊的量词，它加在任何（有效的）正则元素后面都是零宽度匹配，可以匹配任意位置。例如，无论是 x{0} 还是 y{0} ，匹配字符串 abcd ，匹配结果都是：4 个位置（ a, b, c, d 前面的位置以及 d 后面的位置）。为了阅读的便利，这里我们用下划线把匹配到的位置标示出来：_a_b_c_d_。

                                                                  当一个正则表达式是 x* 或 (expr)* 这种形式的时候就要当心了，因为它很有可能会匹配到你不想要的结果。x* 可以看作是 x+|x{0} ，因为是贪婪量词，所以 x{0} 放在第二个分支。如果字符串中不包含 x 字符，那么 x+|x{0} 中的第一个分支 x+ 总会匹配失败、然后尝试第二个分支 x{0}， 总会成功匹配到当前位置——最终结果将会是成功匹配到字符串上的所有位置；如果字符串中包含 x 字符，则会匹配到所有的 x+ (连续的x字符)子串以及不是x前面的任何位置。

                                                                  举个栗子，p* 匹配字符串 apple ，匹配结果为：一个子串 pp ，以及 4 个位置（_app_l_e_）。

                                                                  如果是懒惰量词 x*? 则看作是 x{0}|x+ ，优先匹配最小的次数。p*? 匹配字符串 apple ，结果为： 6 个位置（_a_p_p_l_e_）——没有匹配到任何字符 p ！

                                                                  同理，其他包含零次的可变量词，比如 x? 看作是 x|x{0} 、x{0,5} 看作是 x{1,5}|x{0} 。如果是懒惰形式，则把两个分支顺序调换过来。

                                                                  关于这个特性，最经典的一个误区莫过于使用 .* 匹配“所有字符”。很多人会认为，默认模式下 .* 的匹配结果就是每一行的所有字符；如果开启单行模式，则匹配整个字符串。

                                                                  事实上，默认模式下 .* 的匹配结果是：每一行的所有字符以及行尾的位置 ；单行模式下 .* 的匹配结果是：整个字符串以及字符串最末尾的位置 。

                                                                  举个例子，给定一个字符串 LineOne ，正则查找 .* （默认模式） 替换为 $&后缀 ，直觉上我们会以为结果将会是：LineOne后缀。但实际上却发现结果是：LineOne后缀后缀 ——文本末尾添加了两个“后缀”。这是因为 .* 的匹配结果是：整行文本以及行尾的位置（LineOne_）。第一个匹配项（LineOne） 替换为 $&后缀 ——替换文本是 LineOne后缀、第二个匹配项（行尾位置）替换为 $&后缀 ——替换文本是 后缀，所以最终替换结果为 LineOne后缀后缀 。在线测试

                                                                  从上面的例子可以看到，裸 .* 的匹配结果是比较奇怪的，往往不符合我们预期。如果想要匹配整行内容或整个字符串，应该使用量词 + 而不是 * 。

                                                                  因此，当整个正则表达式可能会产生零宽度匹配项（匹配项是一个位置）的时候，需要留意零宽匹配项是否是自己期待的。比如 ^[0-9]*$ 可以匹配纯数字的字符串，但也可以匹配空字符串。如果空字符串并不符合我们的预期，应该改成 ^[0-9]+$ 。

                                                                  还有一个陷阱就是使用 .* 来匹配剩余内容。比如，我想使用正则匹配以“很久很久以前”开头的文本行，大概会写成 (?m)^很久很久以前.*$ 。这个正则（整个表达式）的匹配项必然会包含字符，匹配项不会是位置，所以不会产生上面的“零宽度匹配项”问题。但是，当目标文本中是以 \r\n 作为行结束符（Line Break）的时候，另一个问题就来了。因为 .NET 正则的 . 匹配的是除了 \n 之外的任意字符，所以 .* 会匹配包含 \r 在内的剩余字符，这个结果并不是我们想要的——期望结果是 "很久很久以前 . . . 乌鸦喝不到水。"，实际上的结果是 "很久很久以前 . . . 乌鸦喝不到水。\r" 。

                                                                  假如我使用这个正则来为文本行添加后缀，正则查找 (?m)^很久很久以前.*$ 替换为 $&后缀 。结果将会是 "很久很久以前 . . . 乌鸦喝不到水。\r后缀" ，插入的后缀把行结束符 \r\n 打断了。为了避免这个情况，可以把 . 换成 [^\r\n] 。

                                                                   

                                                                  锚点 Anchors

                                                                   

                                                                  💡 如何理解锚点？

                                                                  锚点匹配的是特定条件的位置，比如 ^ 匹配的是 位于字符串开头 的位置 。

                                                                  从匹配效果上来说，锚点的作用就是给字符的匹配施加限制条件，相当于把字符的匹配锚定在某个位置上。比如，can可以匹配到字符串 "can you can a can" 中所有的 can ，can 可以出现在字符串的任何位置。但如果在前面添加一个 ^，则等于给表达式 can 的匹配添加一个限制条件：c 前面的位置必须位于字符串开头。这时，就只能匹配到最开头的那个 can 了。

                                                                  这个思维对于理解锚点以及环视是非常有帮助的。 （参考 锚点、环视以及断言）

                                                                   

                                                                   

                                                                  🤔 关于 CR 与 LF 的问题

                                                                  行结束符（Line Breaks）是指文本中用于表示一行结尾的字符或字符序列，在不同的操作系统或应用程序中可能有不同的表示方式。常见的行结束符包括：CR+LF（\r\n）、LF（\n）以及 CR（\r）。

                                                                  当我们使用正则表达式来处理多行的文本内容时，需要在三个层面上关注行结束符：

                                                                  1. 原始文本中的行结束符是哪一种形式？
                                                                  2. 应用程序（编程语言）读取文本的过程是否对行结束符进行了自动转换？
                                                                  3. 正则引擎对行结束符的定义？

                                                                  让我们通过一个例子来阐述这三点内容。

                                                                  首先，我们在 Windows10 系统上打开“记事本”编辑器，依次输入“张三”回车键 “李四”，最后保存为“test.txt”。此时，“test.txt”中的文本内容为“张三\r\n李四”，行结束符是 CR+LF 。

                                                                  接着，让我们使用 EmEditor 编辑器来打开“test.txt”，按下 Ctrl + H 进行正则替换操作。此时，我们（或者说正则引擎）面对的文本内容依然是“张三\r\n李四”吗？不是。EmEditor 读取文本文件时，会自动识别各种行结束符并统一转换为 LF，此时我们面对的文本是“张三\n李四”，进行查找替换时并不需要考虑 \r 字符。

                                                                  很多编辑器或编程语言在读写文本方面都遵循这样的策略：读取文本文件时，默认会自动识别不同的行结束符，将其统一为 \n 进行处理；写入文本文件时，默认根据当前操作系统的约定使用适当的行结束符。

                                                                  在这种情况下，最终正则引擎面对的文本都是以 \n 作为行结束符的，正则处理并不存在“方言问题”，因为所有正则引擎都会把 \n 视为行结束符。

                                                                  但如果没有这个自动转换过程，“test.txt”的内容以最原始的形式传递到正则引擎呢？此时就得考虑“方言问题”了，因为不同的正则引擎对“行结束符”的定义是有差异的。例如，.NET 正则引擎定义的行结束符只有一种：\n ；Javascript 正则引擎定义的行结束符为：\n、\r、\u2028、\u2029 ；Java 正则引擎定义的行结束符为：\r\n、\n、\r、\u0085、\u2028、\u2029；……

                                                                  不同的正则引擎之间对行结束符的定义有差异，相应地，与文本行相关的正则语法（如 . ^ $ 等）自然也就存在“方言问题”了。

                                                                  以字符串“张三\r\n李四”为例：

                                                                  • 在 .NET 正则引擎眼中，第一行内容为“张三\r”，第二行内容为“李四”；
                                                                  • 在 Javascript 正则引擎眼中，第一行内容为“张三”，第二行内容为空字符串“”，第三行内容为“李四”；
                                                                  • 在 Java 正则引擎眼中，第一行内容为“张三”，第二行内容为“李四”。

                                                                  那么，多行模式下的 $ ，匹配的行尾（位置）也不尽相同（以下划线标示位置）。

                                                                  • 【 .NET】->"张三\r_\n李四_"
                                                                  • 【Javascript】->"张三_\r_\n李四_"
                                                                  • 【Java】->"张三_\r\n李四_"

                                                                  C# 用户容易踩坑的一个地方就是使用 $ 来匹配行尾字符。例如，(?m)张三$ 凭直觉应该是可以成功匹配到行尾的“张三”的，但实际测试却找不到匹配，为什么呢？因为第一行的内容是“张三\r”，“张三”后面还有一个“\r”字符，并没有紧挨着行尾。为了兼容这种情况，正则表达式应该写作 (?m)张三(?=\r?$) 。

                                                                  值得注意的是，regex101 测试网站并不支持设置行结束符，贴进去的测试文本，行结束符会统一转换为 \n 。而 RegexBuddy 则支持设置测试文本的行结束符。

                                                                  还有更多的与行结束符相关的语法，不同的正则引擎之间表现有所差异，就不一一赘述了。

                                                                   

                                                                   

                                                                  环视 Lookarounds

                                                                   

                                                                   

                                                                  📖 环视的术语解释

                                                                  环视有四种：

                                                                  • (?=expr) 肯定型前视（positive lookahead）
                                                                  • (?!expr) 否定型前视（negative lookahead）
                                                                  • (?<=expr) 肯定型后视（positive lookbehind）
                                                                  • (?<!expr) 否定型后视（negative lookbehind）

                                                                  (?...) 是正则扩展语法的固定结构，问号后面跟随不同的字符构成不同的语法，比如跟着 : 是非捕获组 (?:expr)、跟着 > 是原子组 (?>expr) 等等。

                                                                  在环视的语法中，= 和 ! 分别就是 匹配 和 不匹配 的意思：(?=expr) 表示当前位置可以匹配 expr ，(?!expr) 表示当前位置不能匹配 expr ——这两个语法的匹配方向与正则引擎默认的匹配方向一致，即向右匹配。而在 = 或 ! 前面加上 < ，则变成向左匹配的版本了：(?<=expr) 表示当前位置向左可以匹配 expr ，(?<!expr) 表示当前位置向左不能匹配 expr 。

                                                                  (?=expr) (?!expr) 被称作 lookahead 、(?<=expr) (?<!expr) 被称作 lookbebind ，这两个术语字面上就是“向前看（查找）”、“向后看（查找）”的意思，对应着子表达式 expr 的匹配方向——向右匹配的叫 lookahead ，向左匹配的叫 lookbehind 。lookahead 和 lookbehind 统称为 lookaround 。

                                                                  (?=expr) (?<=expr) 要求子表达式 expr 能够匹配，而 (?!expr) (?<!expr) 则要求子表达式 expr 不匹配，所以我们说前者是肯定型的（positive），后者是否定型的（negative）。

                                                                  这几个环视的名称在国内的翻译极其不统一，可谓五花八门，像什么“零宽度负预测先行断言”之类的，听着就令人望而生畏。希望这番术语解释能帮助到你更好地理解和记忆这个语法。

                                                                   

                                                                  📖 什么是断言？

                                                                  诸如 a \n . [abc] 这一类正则元素，它们的作用都是匹配单个字符。而锚点和环视则是很不一样的另一种类型：它匹配的不是字符，而是位置。它的作用是在匹配过程中测试一个位置是否符合给定条件，只返回匹配结果：“匹配”或“不匹配”。正因如此，这一类正则元素被称为“断言”（assertion）。又因为断言的匹配不会返回任何字符，其匹配结果的长度为零，所以也叫“零长断言”（zero-length assertions）或“零宽断言”（zero-width assertions）。

                                                                  🚩 断言包括锚点（anchor）和环视（lookaround）。

                                                                   

                                                                  💡 锚点、环视以及断言

                                                                  字符类匹配的是某个字符集合中的任意一个字符，这个字符集可以是正则引擎预定义的（如 . \d \p{Sc}），也可以是用户自定义的（如 [abc]）。

                                                                  类似地，我们也可以这样理解锚点和环视：两者都是匹配 特定位置 的断言，锚点是预定义的断言，而环视则是自定义的断言。

                                                                  锚点匹配什么样的位置，是正则引擎预先定义好的。例如，^ 匹配的是 位于字符串开头 的位置，\z 匹配的是 位于字符串末尾 的位置……

                                                                  而环视匹配什么样的位置，这个条件完全是由用户通过子表达式 expr 来自定义的。例如，(?=expr)匹配的是 右边是 expr 的位置，(?<!expr)匹配的是 左边不是 expr 的位置……正是这个高自由度特性，使环视成为正则最强大的功能之一。

                                                                   

                                                                   

                                                                  选择分支 Alternation

                                                                   

                                                                  🤔 选择分支的优先级

                                                                  一个简单的例子，相信很多初学者都有过这样的疑惑：正则 a|bc 的含义到底是 a 或 bc ，还是 a 或 b 然后 c ？

                                                                  换个说法就是， a|bc 的匹配结果会等同于 (a)|(bc) 还是 (a|b)c ？答案是前者，因为在所有正则操作符中，竖线 | 的优先级是最低的，这个正则的两个选择分支分别是 a 和 bc 。如果希望两个分支分别是 a 和 b ，那么可以通过使用括号分组来实现：(a|b)c 。这个做法类似于四则运算把 1+2*3 改成 (1+2)*3 ，通过优先级最高的括号改变运算结构。

                                                                  再来看一个例子。我们都知道，如果想要匹配一个完整的单词，需要使用词边界 \b 来限制匹配，例如 \bTom\b 。那假如我想要匹配 Tom, Dick, Harry 三个单词中的任意一个呢？一个常见的错误写法是 \bTom|Dick|Harry\b 。这个正则的三个选择分支分别是 \bTom、Dick、Harry\b ，并没有让每个单词都完整地受到 \b 约束，\bTom|Dick|Harry\b 是可以匹配到 Tomorrow 中的 Tom 的。正确的写法是 \b(Tom|Dick|Harry)\b 。

                                                                  最后，来个练习题：

                                                                  正则 ^hello|world$ ，等同于下面哪一条正则？

                                                                  ^hell(o|w)orld$

                                                                  ^(hello|world)$

                                                                  (^hello)|(word$)

                                                                  如果我想匹配完整内容为 "hello" 或 "world" 的字符串，应该使用哪一条正则？

                                                                  🤔 后向引用与子表达式引用

                                                                  (\d+)-\1 与 (\d+)-\d+ 是否等价？

                                                                  对后向引用的一个常见误解就是，认为(\d+)-\1 是等同于 (\d+)-\d+ 的。

                                                                  (\d+)-\1 中的 \1 是对第一个捕获组的引用，但它引用的是捕获组的确切内容，而不是捕获组里面的表达式！

                                                                  例如，如果第一个捕获组匹配了数字 123，那么 \1 将与 123 匹配。(\d+)-\1 可以匹配字符串 123-123，但不能匹配字符串 123-456 。

                                                                  那么，有没有可以引用捕获组表达式的语法呢？答案是有的。例如，在 Perl 正则表达式中，可以使用 (?1) 来引用第一个捕获组的子表达式，(\d+)-(?1) 可以匹配 123-456，等同于 (\d+)-\d+ 。

                                                                  不过，支持子表达式调用的正则引擎有限，C#、Java、Javascript、Python 等主流语言的标准正则表达式库都不支持该语法。

                                                                  🤔 单行模式与多行模式

                                                                  单行模式（singleline）和多行模式（multiline）绝对是最容易让人迷惑的一对正则术语。这两个名词，如果孤立地审视为什么这样命名，似乎不无道理——但它们的同时存在，则堪称灾难了。别说正则初学者，即使是正则老手，误解这两个概念的也大有人在。

                                                                  以下是几个常见的可能误解：

                                                                  • 单行模式和多行模式是一对互斥概念，不能同时使用。
                                                                  • . 可以跨行匹配的时候是多行模式，反之则是单行模式。
                                                                  • ^ $ 仅匹配整个字符串的开头/结尾时是单行模式，匹配每行的开头/结尾时是多行模式。
                                                                  • 单行模式下，输入文本被视为一个字符串，没有多行的概念，^ $ 匹配整个字符串的开头/结尾，. 可以匹配包括换行符在内的任何字符； 多行模式下，输入文本被视为多个字符串，正则单独对每一行进行匹配，^ $ 匹配每行的开头/结尾，. 不能匹配换行符；

                                                                  事实上，“单行模式”和“多行模式”两个选项之间没有任何关系。“单行模式”只会影响 . 的匹配，“多行模式”只会影响 ^ $ 的匹配，它们之间毫无关联、互不干扰，是两个相互独立的选项开关。

                                                                  . 的默认含义是“匹配除行结束符外的任意单个字符”。如果你把“单行模式”这个选项勾上，那么它的含义将被修改为“匹配包括行结束符在内的任意单个字符”。无论你是否勾选“多行模式”，都不会影响 . 的含义。

                                                                  ^ $ 的默认含义分别是“匹配整个字符串的开头/结尾”。如果你把“多行模式”这个选项勾上，那么它俩的含义将被修改为“匹配每行的开头/结尾”。无论你是否勾选“单行模式”，都不会影响 ^ $ 的含义。

                                                                  为了避免混淆，很多编程语言开始摒弃“单行模式”这个叫法，改叫 dot all 模式（点号通配之意）。RegexBuddy 更是干脆，直接把这两个选项分别叫做“Dot matches line breaks”和“^$ match at line breaks”，即“点号匹配行结束符”和“^$ 匹配换行处”。

                                                                  Match 方法

                                                                  示例

                                                                  获取的第一个匹配项

                                                                  xxxxxxxxxx
                                                                  string text = "The Fat Cat Sat on the Mat";
                                                                  string pattern = @"\b.at\b";
                                                                  

                                                                  Match m = Regex.Match(text, pattern);
                                                                  

                                                                  if (m.Success)  // 如果找到匹配项
                                                                  {
                                                                      Console.WriteLine(m.Value);  // Fat
                                                                  }

                                                                   

                                                                  遍历所有匹配

                                                                  xxxxxxxxxx
                                                                  string text = "The Fat Cat Sat on the Mat";
                                                                  string rx = new Regex(@"\b.at\b");
                                                                  

                                                                  Match m = rx.Match(text);
                                                                  

                                                                  while (m.Success)
                                                                  {
                                                                      Console.WriteLine(m.Value);
                                                                      m = m.NextMatch();
                                                                  }

                                                                   

                                                                  获取匹配项的分组内容

                                                                  xxxxxxxxxx
                                                                  string text = @"C:\Users\Tom\小说\红楼梦.txt";
                                                                  string pattern = @"小说\\(.+?)\.txt$";  // 分组1匹配的是小说名称
                                                                  

                                                                  Match m = Regex.Match(text, pattern);
                                                                  

                                                                  if (m.Success)
                                                                  {
                                                                      // 读取匹配项的分组1的值
                                                                      string name = m.Groups[1].Value;
                                                                      Console.WriteLine(name);
                                                                  }

                                                                  Escape 方法

                                                                  对正则表达式中的 \ * + ? | { [ ( ) ^ $ . # 以及空白字符进行转义，按字面意思匹配这些字符本身。也就是说，Escape 方法会转义正则表达式中的所有元字符，转义后的表达式将会匹配在字面上与原表达式一致的字符串。

                                                                  示例

                                                                  xxxxxxxxxx
                                                                  string text = "11121+2";
                                                                  string pattern = @"1+2";  
                                                                  string escapedPatt = Regex.Escape(pattern);  // 1\+2 
                                                                  

                                                                  string result1 = Regex.Match(text, pattern).Value;  // 1112
                                                                  string result2 = Regex.Match(text, escapedPatt).Value;  // 1+2
